# Anydocs 共享设计系统

这份文档定义 Anydocs 的共享视觉语言。它不是某个单独页面的样式表，而是 Reader、Studio 和未来 surface 的共同基线。

## 设计定位

Anydocs 是本地优先的文档编辑器和阅读站，视觉上应该体现以下特征：

- 产品级，而不是内容平台感
- 克制、清晰、稳定
- 偏工具，不偏营销
- 让阅读和写作都更快，不靠装饰分散注意力

## 设计原则

### 1. 少颜色，多语义

- 全局主色只保留 1 个
- 语义色只在成功、警告、错误、信息上出现
- 大多数界面由中性色和边框承担结构
- 默认不要使用 indigo / violet 作为品牌色

### 1.1 共享 token，不共享 chrome

Anydocs 只共享底层视觉语义，不共享完整界面壳层。

这意味着：

- Reader 和 Studio 可以共用颜色、字体、间距、圆角、动效 token
- Reader 和 Studio 不能共用同一套布局 chrome
- Reader 的不同主题可以共享 token 规则，但允许有不同的导航、页头、侧栏和内容宽度
- Studio 需要更高的信息密度和更强的操作反馈，不能直接复用 Reader 的阅读壳层

换句话说，统一的是“语言”，不是“页面骨架”。

推荐语义色：

| 角色 | 建议值 |
| --- | --- |
| 主色 | `#2563eb` |
| 主色 hover | `#1d4ed8` |
| 主色浅底 | `#eff6ff` |
| 成功 | `#16a34a` |
| 警告 | `#d97706` |
| 错误 | `#dc2626` |
| 信息 | `#0ea5e9` |

### 2. 少字体，多权重

- UI 字体优先用 `Inter`
- 等宽字体优先用 `IBM Plex Mono`
- 一个界面里不要超过 2 套字体
- 通过字号、权重、行高建立层级，而不是到处换字体

推荐字号：

| 用途 | 大小 |
| --- | --- |
| Caption / 辅助文本 | 12px |
| Meta / 标签 | 13px |
| UI 正文 | 14px |
| 普通正文 | 16px |
| 小标题 | 20px |
| 页面标题 | 28px - 32px |
| Hero / 特殊标题 | 40px+ |

推荐权重：

- 400: 正文
- 500: 标签、导航、轻量强调
- 600: 小标题、按钮
- 700: 页面标题、强强调

### 3. 少动效，多反馈

- 高频操作只保留轻微反馈
- 只有状态变化、抽屉、弹层、排序时才使用明显动效
- 动效服务于连续性和反馈，不服务于炫技

推荐时长：

| 场景 | 时长 |
| --- | --- |
| Hover / Press | 90ms - 120ms |
| Tab / 小面板 | 160ms - 220ms |
| Drawer / Modal | 220ms - 320ms |

### 4. 少装饰，多层级

- 边框、阴影、背景层次优先于花哨渐变
- 卡片只在需要分区时才使用
- 信息密度高的界面要更克制，不要为了“轻盈”牺牲可读性

## 基础 token

### 色彩

建议的浅色主题基线：

| Token | 建议值 |
| --- | --- |
| `bg` | `#fafaf8` |
| `surface` | `#ffffff` |
| `surface-2` | `#f8fafc` |
| `border` | `#e5e7eb` |
| `border-strong` | `#cbd5e1` |
| `text` | `#111827` |
| `text-muted` | `#4b5563` |
| `text-subtle` | `#6b7280` |
| `text-inverse` | `#ffffff` |

### 圆角

| Token | 建议值 |
| --- | --- |
| `radius-sm` | 6px |
| `radius-md` | 10px |
| `radius-lg` | 14px |
| `radius-xl` | 18px |

### 间距

推荐使用 4px 基线：

`4, 8, 12, 16, 20, 24, 32, 40, 48`

### 阴影

- 仅保留 1 - 2 级阴影
- 阴影要轻，尽量像“分层”而不是“漂浮”

推荐：

- `shadow-sm`: 轻描边感
- `shadow-md`: 少量 elevation，用于面板和弹层

## 组件规则

### Button

按钮只保留 4 类：

- Primary: 主动作
- Secondary: 次动作
- Ghost: 导航或低干扰操作
- Destructive: 删除、危险操作

按钮要求：

- 高度统一
- 文案短而明确
- Primary 永远是最强对比
- 不要用渐变按钮作为主 CTA

### Input

- 输入框边框要清楚
- Focus 时用主色 ring 或边框强调
- 错误状态不要只换颜色，最好同时给出文本提示

### Card

- 卡片用于分组，不用于装饰
- 卡片之间靠间距建立层级
- 只有内容区需要分离时才加边框和阴影

### Sidebar

- 左侧导航负责“定位”
- 激活态必须一眼可见
- 导航项不要靠太多图标表达层级

### Tabs

- 用于状态切换，不用于导航大分流
- 激活态应通过颜色、下划线或底色三选一，不要全用

### Status Chips

- 用于 draft / in review / published / warning / success
- 颜色要浅，文字要明确
- 不要让 chip 成为页面里最吵的元素

## 内容规则

### Reader 文本

- 正文行高建议 1.55 - 1.7
- 阅读列宽度建议 65 - 80 个字符
- 标题和正文之间要保留明显空间
- 代码块需要比正文更稳，不要抢视觉

### Studio 文本

- 标签和字段名要短
- 帮助文案只补充必要信息
- 空状态文案要明确下一步，不要写得像营销页

## 统一的反模式

- 默认紫色 / indigo 主题
- 所有页面都像 landing page
- 卡片过多、边框过多、阴影过多
- 文案很软但动作不明确
- 图标堆砌代替层级设计
- 颜色很多，但没有语义

## 与现有配置的关系

这套设计系统最终应该映射到项目配置里的这些区域：

- `site.theme.branding`
- `site.theme.chrome`
- `site.theme.colors`
- `site.theme.codeTheme`

也就是说，设计系统负责“应该长什么样”，主题配置负责“把这个样子落到项目里”。

## Surface 边界

- Reader 是公开阅读 surface，允许存在多个主题实现
- Studio 是私有编辑 surface，强调效率和管理，不追求与 Reader 同形
- 同一个项目可以在 Reader 上使用不同主题，但这些主题的 chrome 可以不同
- 不要把 Reader 的主题当成 Studio 的主题，反过来也一样
