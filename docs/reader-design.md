# Reader DESIGN.md

这是 Anydocs 阅读站的 surface 规范。它定义的是“读起来像一个产品”该怎么做，不是单纯的页面布局说明。

## Reader 的目标

- 让用户快速找到内容
- 让正文阅读舒适、连续、可信
- 让代码、步骤、导航、搜索都服务于“理解”
- 像成熟的开发者文档产品，而不是博客或知识库集合
- Reader 允许存在多个主题，且这些主题不需要互相长得一样

## 参考方向

Reader 的视觉基线建议参考：

- Cursor Docs 的克制阅读感
- Raycast Docs 的层级、留白和目录感

共同特征：

- 左侧固定导航
- 中间阅读列清晰、稳定
- 右侧只保留必要的上下文信息
- 强调内容结构，不强调装饰

## 主题原则

Reader 的主题不是简单换色。

每个主题都可以在以下方面做自己的决定：

- 页头是否存在
- 左侧导航是否更紧凑
- 右侧 rail 是否启用
- 内容列宽度
- 搜索入口位置
- 品牌区呈现方式

但主题不应该破坏 Reader 的本质：

- 阅读优先
- 目录可扫视
- 代码可复制
- 内容连续

也就是说，Reader 的不同主题可以“像不同产品”，但不能失去“像文档站”的共同底色。

## 结构

### Desktop

推荐的桌面布局是三段式：

1. 左侧导航
2. 中央阅读内容
3. 右侧上下文 rail

尺寸建议：

- 左侧导航：240px 左右
- 中央内容：760px - 880px
- 右侧 rail：200px - 240px
- 列间距：24px - 32px

### Header

头部只保留必要元素：

- 品牌标题或 logo
- 搜索
- 语言切换
- 少量工具入口

不要把 Reader 做成一个功能密度很高的控制台。头部的责任是“定位”，不是“占空间”。

### Sidebar

左侧导航规则：

- 分组明确
- 当前页清晰高亮
- 支持 sticky
- 搜索优先于滚动浏览

导航不应该喧宾夺主。它要稳定、可扫视、可预测。

### Content Column

内容列是 Reader 的主角。

规则：

- 标题要比正文更稳、更重
- 正文宽度不要过宽
- 段落之间保留足够间距
- 代码块要和正文区分开，但不要太重

推荐层级：

| 元素 | 建议 |
| --- | --- |
| H1 | 28px - 32px，600 - 700 |
| H2 | 20px - 24px，600 |
| H3 | 18px - 20px，600 |
| Body | 16px，400，line-height 1.6 |
| Meta | 13px - 14px，400 |
| Code | 13px - 14px，IBM Plex Mono |

### Right Rail

右侧栏只放真正有帮助的内容：

- 本页目录
- 关键链接
- 上一页 / 下一页
- 相关说明

不要堆太多信息，否则 Reader 会失去“阅读站”的舒适感。

## 内容块

### Code Block

代码块应该：

- 有清楚的边界
- 行高略紧凑
- 支持复制
- 语法高亮克制
- 不用过于花哨的背景效果

### Callout

Callout 只用 3 - 4 种状态：

- info
- success
- warning
- danger

每个 callout 的信息密度都要高，避免写成空泛提示。

### Lists

列表是 Reader 的高频内容：

- 列表项要有稳定缩进
- 编号列表要清楚地表达步骤顺序
- 复杂列表里不要让间距过大

### Next / Prev

如果内容是连续章节，提供 Next / Prev 很重要。

规则：

- 只能在真正存在前后关系时出现
- 样式要轻
- 不要像营销站的 CTA 卡

## 视觉语言

### 推荐基线

- 背景：温和白，不要纯白刺眼
- 文本：深灰，不要纯黑
- 边框：浅灰
- 强调色：蓝色优先，避免紫色默认化
- 图像：只在确实有帮助时使用

### 不推荐

- 大渐变 hero
- 大量浮层装饰
- 过度卡片化
- 过亮的品牌色充满全站

## 交互

### Search

搜索应满足：

- 可见
- 快速
- 结果可读
- 能区分文档、章节、页面

### Navigation

- 点击导航要有明显反馈
- 当前章节要随滚动保持可见
- 移动端要收纳到 drawer

### Motion

Reader 的动效只保留：

- 导航切换
- 搜索结果展开
- Drawer / modal 打开
- 章节折叠

其余一律短、轻、少。

## Mobile

移动端的 Reader 规则：

- 导航折叠为 drawer
- 内容列单列展示
- 右侧 rail 下沉到底部或隐藏
- 代码块横向滚动要可控

## 读写分离

Reader 不应该混入 Studio 的编辑逻辑。

也就是说：

- 读页面时要更安静
- 编辑入口应该轻，不抢内容层级
- 不要在 Reader 里再放一套复杂的工作台 UI

## 验收标准

如果 Reader 做对了，它应该像这样：

- 一眼能看出这是文档产品
- 浏览、搜索、阅读都顺手
- 页面看上去专业但不冷漠
- 内容优先，UI 只是帮助阅读
