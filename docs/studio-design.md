# Studio DESIGN.md

这是 Anydocs Studio 的 surface 规范。Studio 的角色不是展示内容，而是让内容、导航和项目配置可以高效地被编辑和管理。

## Studio 的目标

- 让用户快速完成编辑、保存、预览、发布
- 让页面、导航、设置、状态一眼可见
- 让编辑器像工作台，不像普通表单页
- 让高频动作尽量少跳转、少阻塞、少猜测
- Studio 不需要和 Reader 长得一样

## 参考方向

Studio 的视觉基线建议参考：

- Shopify Admin 的结构清晰和表单稳定性
- Stripe 的设置侧边抽屉和状态提示
- Gladia 那种“代码 + 配置”并列的工作台感

共同特征：

- 左侧导航明确
- 中央编辑区稳定
- 右侧设置区可控
- 动作按钮层级明确

## 与 Reader 的边界

Studio 和 Reader 共享 token，但不共享 chrome。

Studio 应该明显偏向：

- 更高的信息密度
- 更多的状态提示
- 更强的编辑反馈
- 更直接的配置面板

不要把 Reader 的页头、侧栏、内容列直接搬到 Studio。
Reader 主题再好，也不能当作 Studio 的壳层模板。

## 结构

### Desktop Shell

推荐桌面布局为三段式：

1. 左侧导航与项目区域
2. 中央主编辑区
3. 右侧设置 / 元数据 / workflow 面板

尺寸建议：

- 左侧导航：240px - 280px
- 右侧设置面板：320px - 360px
- 中央编辑区：自适应剩余宽度
- 面板间距：24px

### Top Bar

顶部栏应该承担这些职责：

- 当前项目识别
- 保存状态
- 预览 / 构建
- 主要操作入口

顶部栏不应该塞太多二级功能。二级功能进侧边栏或设置面板。

### Left Navigation

左侧导航负责：

- 切换页面
- 切换导航结构
- 切换语言
- 进入项目设置

规则：

- 当前选中项必须清楚
- 支持分组
- 支持拖拽或排序反馈
- 支持折叠，但不要过度隐藏

### Main Editor

编辑区是 Studio 的核心。

规则：

- 编辑器需要足够宽
- 文本、代码、表格、图片块都要有稳定的输入反馈
- 编辑状态不应该频繁跳动
- 选中状态、拖拽状态、保存状态要明确

编辑区不是阅读区。它需要更高密度、更清楚的边界和更强的操作提示。

### Right Settings Panel

右侧面板负责：

- 页面 metadata
- 模板选择
- 发布状态
- 项目主题和品牌设置
- 导航结构相关设置

规则：

- 以分组卡片呈现
- 每组有清晰标题
- field label 短而明确
- helper 文案只放必要信息

## 关键状态

### Page Status

Studio 必须让这些状态清楚可见：

- draft
- in_review
- published

状态应该同时出现在：

- 页面列表
- 编辑页顶部
- 相关 workflow 提示中

### Save State

保存状态必须明确：

- 未保存
- 保存中
- 已保存
- 保存失败

不要只靠一个 spinner 表示一切。用户需要知道自己现在处于哪一步。

### Validation

表单验证规则：

- 错误要贴近字段
- 错误文案要说明怎么修
- 不要只有红色边框，没有解释

### Preview / Build

预览和构建是 Studio 的高频动作。

规则：

- 入口可见
- 状态可见
- 结果可回溯
- 失败时给出明确原因

## 组件规则

### Navigation Composer

- 适合展示层级、拖拽排序和分组
- 激活态必须明显
- 对长列表做局部虚拟化或折叠，避免界面发散

### Page Editor

- 页面内容编辑优先
- 元信息编辑次之
- 不要把页面设置拆得过碎

### Project Settings

- 项目设置应该更像“配置面板”，而不是“表单海”
- 同一组的字段尽量聚在一起
- 颜色、品牌、chrome、codeTheme 适合集中管理

## 视觉语言

### 推荐基线

- 背景：很浅的暖灰或中性白
- 卡片：白底，轻边框
- 主动作：蓝色优先
- 状态色：保守、语义化
- 图标：线性、克制

### 不推荐

- 过度拟物化
- 过多渐变卡片
- 大面积深色控制台风格
- 把 Studio 做成 Reader 的变体

## Motion

Studio 的动效可以比 Reader 稍多，但仍然要克制：

- 抽屉打开：220ms - 320ms
- 面板切换：160ms - 220ms
- 保存反馈：120ms - 180ms
- 拖拽排序：连续反馈优先，不要花哨

## 编辑体验原则

- 高频操作少层级
- 低频操作可进设置
- 用户不应该为了改一个项目设置来回跳 3 次
- 页面编辑和导航编辑要能在同一工作流里完成

## 移动端

Studio 的移动端只保留最必要的读写能力：

- 项目切换
- 页面列表
- 基础编辑
- 基础设置

复杂的导航管理和大段内容编辑仍然优先在桌面完成。

## 验收标准

如果 Studio 做对了，它应该像这样：

- 打开后像一个可靠的工作台
- 页面、导航、设置、发布状态都找得到
- 高频操作快，不需要猜
- 表单多，但不杂乱
- 编辑区和设置区层次清楚
